🤔 功能是什麼？

使用者可以用按鍵(F1~F7)切換多個terminal，切換到不同的Terminal時只會顯示當前的資料，讓使用者可以同時處理許多任務。

😇 如何實作？

在開始實作以前先分析xv6如何讀取指令並執行，最後顯示於screen。

流程分析

User space

  1. init.c
    在init.c(process 1)會發現會openconsole這個inode，如果不存在則執行mknod建立，並且將1,2 這兩個 file descriptor(fd) 也一併dup給console，接著就fork並透過exec啟動sh，然後process 1則負責回收殭屍process。

  2. sh.c
    在sh執行後首先會確保fd 0,1,2 存在，如果不存在則一樣openconsole直到0,1,2都被開啟為止。接著就無窮迴圈執行下列流程

    1. getcmd(buf, 100)取得指令
    2. 如果有收到完整的指令就fork+exec
    3. wait準備回收由exec建立的子process
    4. 重複上述過程

    而getcmd，基本上就是顯示前綴符號($)並alloc buf，接著呼叫呼叫ulib.c的gets取得keyborad input。需注意gets是屬於blocking method，如果沒有新資料會讓process進入sleeping

  3. ulib.c
    gets目標就是不斷呼叫system call sys_read(0, &c , 1)並每次讀1個char，直到讀到結尾字元(/r,/n)或buf滿了為止，注意這邊的0就是fd 0，也就是console這個inode。

Kernel space

  1. sysfile.c & file.c
    sys_read()有三個arg分別是file descriptor, buffer pointer和buffer size。實際上是直接呼叫fileread，在fileread中要看的type是FD_INODE，因為fd0是透過sys_open建立的，所以預設type就是FD_INODE，fileread呼叫readi，而readi用處是read data from inode。

  2. fs.c
    因為console這個inode pointer的type是T_DEV(sys_mknod建立console時的預設)，所以只看type==T_DEV，而readi檢查ip的major是否有效且devsw[major].read這個function pointer是否存在，如果檢查通過則使用devsw的funtion pointer呼叫read()取得輸入，這個read function pointer實際上就是consoleread()。

  3. console.c
    接著看consoleread，consoleread首先會取得spin lock並開始讀input[] buffer(char[128])，如果讀到換行字元或Ctrl+D則跳出，如果讀滿了也會跳出，但是假如while檢查發現read, write索引在同個位置(代表沒有任何input)，則睡覺(channel input.r)，此時procss放棄console的spin lock並sleeping，直到有人對input.r進行wake。
    在sleeping的過程中如果發生鍵盤中斷，代表使用者key in了資料，則會觸發consoleintr，consoleintr會將資料輸入input[] buffer，
    這時如果發生輸入的字元是\n或ctrl+D或input.e變化，則wakeup(input.r)將所有在等待的consoleread叫起來，最後放釋放spin lock讓叫醒的consoleread可以讀取資料
    consoleread起床發現input.r!=input.w因此跳出迴圈並不斷將資料塞入input[]buffer，直到滿足跳出條件為止，跳出後釋放spin lock並return read() method.

User space

  1. ulib.c
    gets會一次read一個字元直到收到完整的指令為止，跳出後代表getcmd已經收到完整的cmd

  2. sh.c
    sh會對收到的cmd進行parse後執行，並將結果透過printf輸出到screen

  3. printf.c
    基本上大部分user space的輸出都是透過printf，printf指令比較簡單，基本上就是呼叫system call write(fd, &c, 1)，其中c是要顯示在console的buffer，fd則通常是1,2(console)。

Kernel space

  1. sysfile.c & file.c
    sys_write呼叫filewrite(類似fileread)，輸入參數包含file descriptor, buffer pointer和buffer size。filewrite實際上則是呼叫writei。

  2. fs.c
    因為console的ip.type依舊是T_DEV，所以檢查major後呼叫devsw[CONSOLE].write(ip, src, n)。

  3. console.c
    最後一步，取得console的spin lock後將buffer中的字元透過consputc()顯示到螢幕上，這邊有比較特別的是有兩步驟，如下
    1.透過uartputc()將字元列印到screen
    2.透過cgaputc()和維護crt[](對應到CGA Memory)這個buffer來計算要把screen閃爍游標顯示到何處。(https://www.codesd.com/item/what-is-the-difference-between-cgaputc-int-c-uartputc-int-c-constputc-int-c-in-xv6.html)

實作

實作分兩階段，如果第一階段能夠順利完成才會進行第二階段。

第一階段：

假設今天已經有多個sh(以下稱tty)在執行，且其中一個被標示為active，其餘inactive的tty都必須將處理的結果輸出到file，以避免screen顯示inactive的tty output，也就是說必須重導這些inactive tty的fd 1,2。

此外為了保留喚醒的能力，console(fd0)不能被關閉，否則當keyboard interrupt打不到，就無法讓該tty重新接上，但也要避免inactive的tty吃掉console的input buffer(除了切換指令）

因為能夠觸發切換指令的起點是console.c，所以先從此處下手，目前的想法 console 增加下列四個變數與一個table。

  1. active tty的pid(啟動預設是第一個sh的pid)
  2. restore flag(default 0)
  3. deactive tty的pid(default -1)
  4. save flag(default 0)
  5. tty 對應 的process pid table

當keyborad收到切換指令時，會將deactive pid換成目前pid，立起save旗標，然後將active pid依據table換成指定的pid，同時立起restore旗標，最後然後wake所有等input.r的process，這時所有在consoleread睡覺的process會去執行下列檢查

  1. 自己是否是active/deactive的process，如果不是則繼續sleep
  2. 如果自己是deactive的process且save旗標立起，則傳指令(foreground->background)給tty，讓tty去進行重導的操作，同時清除save flag。
  3. 如果自己是active的process且restore旗標立起，則發送指令(background->foreground)給tty，讓sh去重導fd 1,2並clean screen和印出先前留存在file的紀錄，同時清除restore期標完成切換。
  4. 如果自己是active/deactive的process但restore/save flag沒有立起則不做事。

為了滿足上述的操作有以下幾個必需要增加的部分

  1. 在init process中fork多個sh，並依序命名為tty0, tty1…，並建立一個tty對應pid的table。
  2. 每個tty都要建立自己對應的file，作為記錄用。
  3. 在sh中實作將fd1,2重導至file or console的功能，這邊會需要system call dup2()
  4. 修改console.c

dup2 linux reference

第二階段：

如果第一階段順利的話，可能會做另外一種virtual console device來取代file，因為file會印出所有的紀錄而非只有螢幕最後保留的紀錄。假使印到file的紀錄很大量可能會導致問題。

🐻 預期成效

使用者可以藉由功能鍵切換tty進行不同的工作，被切換到的tty會把先前的結果顯示在螢幕上，並允許使用者繼續操作。
